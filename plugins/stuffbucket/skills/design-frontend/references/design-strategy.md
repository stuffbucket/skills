# Design Strategy Reference

## Design Briefs

Define the problem space before designing solutions.

### Brief Structure

1. **Project Overview** -- name, summary, business context, stakeholder
2. **Problem Statement** -- what, who, evidence, consequences
3. **Target Audience** -- primary/secondary users, characteristics, persona references
4. **Goals and Success Criteria** -- design goal, measurable metrics, qualitative indicators
5. **Scope and Constraints** -- in/out of scope, technical/brand/timeline/legal constraints
6. **Context and Inputs** -- research, competitive references, previous attempts
7. **Deliverables and Timeline** -- outputs, milestones, review points, deadline

### Checklist

- [ ] Problem-focused, not solution-prescriptive
- [ ] Includes measurable success criteria
- [ ] Stakeholder sign-off obtained before work begins
- [ ] Referenced throughout the project, not filed away

## Design Principles

Principles that genuinely guide decisions, not wall decoration.

### Qualities of Strong Principles

- **Opinionated** -- takes a clear stance
- **Actionable** -- resolves debates
- **Memorable** -- short enough to recall
- **Distinctive** -- reflects this product's values (not generic)
- **Testable** -- designs can be evaluated against it
- **Prioritized** -- rank-ordered so conflicts have a tiebreaker

### Principle Template

For each principle provide:

- Title (3-6 words)
- Statement
- Rationale
- Application example
- Counter-example (what violating it looks like)
- Trade-off acknowledged

### Creation Process

1. Gather inputs (research, values, strategy)
2. Workshop to surface candidates
3. Draft and debate -- test with: "Would anyone disagree with this?"
4. Prioritize for conflict resolution
5. Test against past decisions
6. Socialize widely; display in team spaces
7. Revisit as the product evolves

## North Star Vision

A compelling future state that inspires alignment and guides trade-offs.

### Components

- **Vision Statement** -- who, what experience, why it matters (one sentence)
- **Design Pillars** -- 3-5 strategic focus areas defining differentiators
- **Vision Scenarios** -- concrete narrative stories of the future experience
- **Success Criteria** -- qualitative signals, quantitative metrics, milestones

### Time Horizons

| Horizon | Scope |
| --------- | ------- |
| Near-term (1 yr) | Tangible improvements |
| Mid-term (2-3 yr) | Major experience shifts |
| Long-term (5+ yr) | Aspirational transformation |

### Process

Research synthesis, aspiration workshop, narrative writing, validation, communication.

### Checklist

- [ ] Inspiring but grounded in real user needs
- [ ] Broad enough to accommodate unknowns
- [ ] Actively used in reviews and planning
- [ ] Connected to daily work through pillars

## Competitive Analysis

Systematically evaluate competitor UX to find patterns, gaps, and opportunities.

### Competitor Categories

- **Direct** -- same problem, same audience
- **Indirect** -- same problem, different audience
- **Aspirational benchmarks** -- best-in-class from adjacent domains

### Evaluation Dimensions

Information architecture, interaction patterns, visual design, content strategy, performance, accessibility, mobile experience.

### Feature Comparison Matrix

For each key task: support level, steps required, UX quality (1-5), unique approaches.

### Per Competitor, Document

- Strengths -- what they excel at
- Weaknesses -- friction points
- Table stakes -- patterns users now expect
- Gaps -- unaddressed needs

### Deliverables

Summary overview, comparison matrix, competitor profiles, opportunity map, annotated screenshots.

### Rules

- Analyze full journeys, not isolated screens
- Rate UX quality, not just feature presence
- Include aspirational examples from outside the category
- Update regularly as competitors evolve

## Opportunity Framework

Move from a pile of possible improvements to a prioritized roadmap.

### Opportunity Sources

Research findings, analytics, competitive gaps, technology shifts, stakeholder requests, support channels.

### Evaluation Frameworks

#### Impact-Effort Matrix (2x2)

| | Low Effort | High Effort |
| -- | ----------- | ------------- |
| **High Impact** | Quick wins -- do first | Strategic bets -- plan carefully |
| **Low Impact** | Fill-ins -- do if easy | Deprioritize |

#### RICE Scoring

- **Reach** -- how many users affected per time period
- **Impact** -- magnitude of effect (1-3 scale)
- **Confidence** -- certainty of estimates (percentage)
- **Effort** -- person-weeks to implement

#### Kano Model

- Must-be -- expected; absence causes dissatisfaction
- One-dimensional -- satisfaction scales linearly with fulfillment
- Attractive -- unexpected delight; absence not noticed
- Indifferent -- no effect either way
- Reverse -- presence causes dissatisfaction

#### Value vs. Complexity

Score user value (1-10) and implementation complexity (1-10). Plot and prioritize.

### Output

Ranked list with rationale, theme groupings, dependencies, and confidence levels.

### Rules

- Use multiple frameworks to triangulate
- Include diverse perspectives in scoring
- Revisit as new information arrives
- Document the "why" behind every prioritization decision

## Experience Maps

Map the full ecosystem of user interactions across touchpoints, channels, and time.

### Structure

#### Horizontal Axis: Phases

Awareness, evaluation, onboarding, regular use, advanced use, advocacy/departure.

#### Vertical Layers

- **User Actions** -- what the user does, key decisions
- **Touchpoints** -- website, app, email, support, community
- **Channels** -- desktop, mobile, in-person, automated vs. human
- **Emotions** -- confidence, frustrations, delight
- **Pain Points** -- friction, confusion, information gaps
- **Opportunities** -- improvements, new touchpoints

#### Ecosystem Relationships

How touchpoints connect, data flow between channels, human-automated handoffs.

### When to Use

- New product planning
- Omnichannel experience evaluation
- Ecosystem gap analysis
- Cross-team alignment

### Rules

- Map current state before designing future state
- Include both digital and physical touchpoints
- Involve cross-org stakeholders
- Validate with research, not assumptions

## UX Metrics Definition

Connect design decisions to measurable outcomes.

### Metric Categories

| Category | Examples |
| ---------- | ---------- |
| **Behavioral** | Task completion rate, time on task, error rate, feature adoption |
| **Attitudinal** | SUS, NPS, CSAT, perceived ease of use |
| **Business** | Conversion, retention, support volume, onboarding completion |
| **Engagement** | DAU/MAU, session duration, feature discovery, return visits |

### HEART Framework (Google)

- **Happiness** -- satisfaction, ease ratings
- **Engagement** -- frequency, depth of use
- **Adoption** -- activation rate, feature uptake
- **Retention** -- return rate, churn
- **Task success** -- completion rate, time, errors

### Metric Definition Template

| Field | Description |
| ------- | ------------- |
| Name | Metric name |
| Definition | Precise calculation |
| Method | How to collect |
| Data source | Where data lives |
| Target | Goal threshold |
| Frequency | How often measured |
| Owner | Who is responsible |

### Rules

- Choose 3-5 primary metrics (no more)
- Balance behavioral and attitudinal measures
- Set baselines before measuring change
- Connect every metric to a design hypothesis
- Report quantitative alongside qualitative insights

## Stakeholder Alignment

Navigate stakeholder landscapes and create alignment around design decisions.

### Alignment Artifacts

#### Stakeholder Map

Identify all stakeholders. Plot on influence vs. interest grid. Categorize roles.

#### RACI Matrix

| Decision | Responsible | Accountable | Consulted | Informed |
| ---------- | ------------- | ------------- | ----------- | ---------- |
| (fill per decision) | | | | |

#### Decision Framework

- What decisions need input
- Who has final authority
- How to resolve disagreements

#### Communication Plan

- Who gets what information, when
- Cadence and channels
- Feedback timelines

#### Feedback Protocol

- Expected format and specificity
- Timing within project lifecycle
- How to prioritize conflicting feedback
- Conflict resolution process

### Common Challenges and Mitigations

| Challenge | Mitigation |
| ----------- | ------------ |
| Stakeholders designing solutions | Redirect to problem framing; present options |
| Conflicting priorities | Refer to ranked principles and success criteria |
| Late-stage scope changes | Point to signed-off brief; assess impact formally |
| Missing stakeholders | Map at kickoff; re-check at each milestone |

### Rules

- Map stakeholders at kickoff
- Establish decision rights before conflict arises
- Communicate proactively, not reactively
- Document all decisions and rationale
- Revisit stakeholder dynamics as projects evolve

## Strategy Workflow

Typical sequencing when building a UX strategy from scratch:

1. **Design brief** -- frame the problem and constraints
2. **Competitive analysis** -- understand the landscape
3. **Experience map** -- map current-state ecosystem
4. **North star vision** -- define the aspirational future
5. **Design principles** -- codify decision-making criteria
6. **Opportunity framework** -- prioritize what to build
7. **Metrics definition** -- define how to measure success
8. **Stakeholder alignment** -- align roles, decisions, communication

Each artifact feeds the next. Revisit earlier artifacts as later work surfaces new information.
